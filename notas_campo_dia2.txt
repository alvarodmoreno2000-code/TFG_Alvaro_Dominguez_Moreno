Observaciones generales:
- La Raspberry Pi queda fija en R01 durante toda la campana.
- El Heltec se desplaza secuencialmente por las posiciones H01, H02, H03 y opcionalmente H04/H05.
- El Heltec no necesita conexion a Internet, solo alimentacion y antena LoRa conectada correctamente.
- La Raspberry si necesita Internet para reenviar las tramas al servidor TTN.
- El nodo Heltec se une a TTN mediante OTAA.
- Una vez unido, no se debe pulsar RST, desconectar alimentacion ni cambiar el cable USB/powerbank.
- El join OTAA se realiza cerca del gateway para asegurar la recepcion del Join Accept.
- La captura valida de cada posicion comienza cuando el Heltec ya esta unido y se encuentra fisicamente en la posicion correspondiente.
- El payload mantiene la etiqueta LORAWAN1,H01,... como identificador del nodo/firmware, pero la posicion real se asigna por el punto de campana y por el nombre del archivo JSON.
- Para el dataset se utilizaran los metadatos de TTN/gateway: RSSI, SNR, data rate, frecuencia, gateway ID/EUI y contador de trama.
- Se considera muestra valida aquella recibida por el gateway propio loragateway-tfgalvaro.
- No se consideran muestras validas los Join Request ni Join Accept.
- Los paquetes con DevAddr 00000000 corresponden a intentos de union OTAA y no deben usarse como muestras de campana.
- Las muestras validas aparecen como uplinks de aplicacion con DevAddr asignado y FCnt creciente.
- Durante la prueba se observo ajuste dinamico del data rate por ADR, pasando de SF9BW125 a SF8BW125 y posteriormente a SF7BW125.
- La exportacion de datos se realiza desde TTN Live data del end device heltec01 mediante Export as JSON.
- Cada posicion debe guardarse con un nombre de archivo independiente para evitar mezclar puntos.

Incidencias y decisiones durante la campana:
- En los primeros intentos desde una posicion mas alejada, el Heltec envio Join Request pero no recibio correctamente el Join Accept.
- Al acercar el Heltec al gateway, se completo correctamente el proceso OTAA.
- Se detecto que pulsar RST reinicia el Heltec y obliga a repetir OTAA, por lo que durante la campana no debe reiniciarse el nodo.
- Para evitar esperas largas, se redujo el intervalo entre uplinks de 60 s a 30 s.
- Tras modificar el intervalo, se verifico en el Serial Monitor el mensaje: Proximo uplink en 30 segundos.
- El error puntual de envio de uplink observado tras una reprogramacion no se mantuvo tras reiniciar y volver a unir correctamente el nodo.
- La campana definitiva se realiza con el Heltec ya unido a TTN y desplazandolo encendido entre posiciones.
- La posicion real de cada muestra se asignara por el archivo JSON y por el punto de campana, aunque el payload mantenga la etiqueta H01.
